# Village Adaptation Guide

How a model built for industrial captive power represents village energy
systems, and how its scenarios map to the questions of the 100 GW village solar
program.

## The core analogy

The framework co-optimises a **grid** (zonal capacity expansion + dispatch) and
a set of **decentralised sites** that have their own demand, their own
generation/storage candidates, and an optional grid connection. For the
original studies the sites were industrial parks; for this program each site is
a **village or village cluster**:

| Model concept | Industrial-park reading | Village reading |
|---------------|------------------------|-----------------|
| Site (`Village` ID) | Industrial park | Village / village cluster (e.g. a *Listrik Desa* cooperative) |
| `village_demand.csv` | Factory electric load | Household + productive-use load |
| `village_demandheat.csv` | Process heat | Usually zero (electricity-only) |
| `village_generators.csv` | Captive coal/gas units | Existing diesel gensets + candidate solar PV and batteries |
| `vVIL_IMPORT` at `import_price` | Industrial grid purchases | Village grid imports after interconnection |

## Scenario semantics

`scenario_settings()` in `functions/preflight.jl` maps scenario names to four
flags — `Grid` (transmission expansion + village imports allowed) and
`VillageBuild` (village-side investment allowed) are the two that matter here:

| Scenario | Grid | VillageBuild | Village reading |
|----------|:---:|:---:|------------------|
| `base` | – | – | Grid-only counterfactual; village demand is **not modelled**¹ |
| `grid` | ✓ | – | Grid expansion; villages stay on existing diesel only |
| `village` | – | ✓ | **Standalone**: each village sizes solar + battery + diesel independently, no interconnection |
| `gridvillage` | ✓ | ✓ | **Coordinated**: village assets, grid expansion, and interconnection co-optimised |
| `highimportprice` | ✓ | ✓ | Coordinated with imports priced ×1.21 |
| `nocoal` | ✓ | – | Village demand served by RE/storage + imports only (no thermal commitment on-site) |

¹ In `base`/`grid` runs the village balance only lets *existing committed
units* (diesel) serve village demand; with `grid` also active, imports help.

The concept note's two preliminary scenarios are therefore: **standalone =
`village`**, **coordinated = `gridvillage`**. The headline result is the
difference in total cost, emissions, and reliability between those two runs on
identical data (see `docs/outputs_guide.md`).

## Things to know before trusting village results

- **RE share counts grid generation only.** Village solar does not contribute
  to the `RE_limit` constraint in `clean` runs.
- **Village `Max_Cap_MW` is not enforced** — village solar/battery power
  capacity is unbounded; storage energy is capped per unit by
  `village_storage_max_mwh` (config key, default 208 MWh). Set realistic caps
  via the config or re-enable the bound in `optimizer.jl` if land/network
  limits matter.
- **`import_price` is a flat $/MWh** — no time-of-day structure, no PPA/tariff
  detail, no export compensation (`vVIL_EXPORT` exists in the code but is
  commented out). Export of surplus village solar to the grid is currently
  **not modelled** — a known limitation for the village solar study.
- **Unit commitment on small gensets**: village diesel with `Commit = 1` gets
  binary start/stop variables per hour. With many villages this dominates solve
  time; consider `Commit = 0` with `Min_Power_MW = 0` for sub-100 kW gensets if
  runs get slow.
- **Reliability** is represented by VOLL + curtailment segments in
  `village_demand.csv`, not by a reserve-margin constraint.

## Worked example

`data_indonesia/2030/timor_demo/` (generated by `tools/make_timor_demo.py`) is
a complete synthetic village setup. On it, `village` retires all four diesel
gensets and builds solar ≈2.5× and battery ≈1.3× each village's peak;
`gridvillage` builds ~12% smaller village assets, imports marginally from the
grid, and is cheaper overall — the expected coordination signature.
